## L-08A-001: AC-6 coverage gate の構造的未達（view-model 経由設計）

- 事実: Phase 11 実測で coverage Statements 84.18% / Branches 84.13% / Functions 83.37% / Lines 84.18% が AC-6（Stmts ≥85%）に 0.82pt 不足。代表 authz matrix と route tests + repository tests + invariants では公開 use-case 直接観測点が薄かった。
- 原因: 公開 4 route（`/`, `/members`, `/members/[id]`, `/register`）は view-model 経由で 04a public API を呼ぶ設計で、route handler 側は thin wrapper となり実行 path が test に直接出にくい。view-model 単体 test が薄い領域は coverage 線が引けない。
- 教訓: 公開 use-case 直接観測点を `apps/api/src/__tests__/` 配下に集約する pattern を **Phase 1 の test 戦略段階で確立**する。view-model 経由設計を採用する場合、Phase 4 test strategy で「view-model 単体 test の coverage 目標」と「route handler は thin wrapper として coverage から除外可能か」を明示判断する。AC-6 設計時に「公開 route + view-model 合算 coverage」など合成 metric を採用する選択肢も検討。
- 適用: 09a/09b では 08a partial を green 前提にせず、AC-6 hardening 経路（UT-08A-01）の進捗とは独立に進める。

## L-08A-002: cross-cutting authz テストの集約と route 委譲の境界

- 事実: 全 endpoint × role × action 網羅 matrix を自動生成しようとしたが、Auth.js セッション fixture 生成と admin/public 境界の組合せが endpoint ごとに微妙に異なり、network/IO mock の 1 ファイル集中は維持コストが高い。
- 教訓: `authz-matrix.test.ts` で **代表 matrix（公開 vs 認証必須 vs admin only の 3 層）を 1 ファイル集約**し、個別 endpoint の細かい認可分岐は既存 route tests に委譲する。集約 ↔ 委譲の境界は「role × resource × action のうち、role 軸の網羅性」で線を引き、resource × action の細かい matrix は route 側で持つ。
- 適用: 新規 endpoint 追加時は (a) `authz-matrix.test.ts` に 3 層所属を追記、(b) 細かい分岐は route test 側で生成、の 2 段階を必須化する。

## L-08A-003: invariants 集約テストの粒度トレードオフ

- 事実: 不変条件 #1 / #2 / #5 / #6 / #7 / #11 を `invariants.test.ts` 1 本に集約 assert したことで、test 命名が `invariant #N` 主導となり個別 invariant の検索性が落ちた。
- 教訓: invariants 集約は採用するが、`describe` ブロックを `describe('invariant #1: <短い意味)', () => ...)` のように invariant ID と意味を 1:1 マッピングする命名規約で検索性を担保する。invariants の意味は `outputs/phase-12/phase12-task-spec-compliance-check.md` の trace 表と同期する。
- 適用: 新規 invariant 追加時は (a) `invariants.test.ts` に describe 追加、(b) `phase12-task-spec-compliance-check.md` に trace 行追加、を同一 PR で実施。

## L-08A-004: brand 型の runtime 観測手段の必要性

- 事実: `asResponseId` / `asResponseEmail` 等の brand 型は TypeScript の型安全性を担保するが、runtime での誤値投入（特に zod 経由の境界外）に対する検出は別途必要。
- 教訓: brand 型は型安全だけでなく **runtime 健全性 test を別ファイル（`brand-type.test.ts`）で観測**する。型 narrow 関数 (`isResponseId` 等) と zod refine が validation 失敗時に同一 error code を返すかも検証対象に含める。
- 適用: 新規 brand 型追加時は `brand-type.test.ts` に runtime test を必ず追加。`packages/shared` 側の `@ts-expect-error` 型 test は UT-08A-05 で別途整備する。
