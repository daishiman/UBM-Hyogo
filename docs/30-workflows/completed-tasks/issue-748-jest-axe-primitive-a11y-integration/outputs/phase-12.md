# Phase 12 — implementation-guide（中学生レベル概念説明）

[実装区分: 実装仕様書]

## 12.1 一言で

「ボタンや入力欄が、目が見えない人にも使いやすい形で作られているか」を、毎回テストするたびに機械が自動でチェックしてくれる仕組みを入れる作業です。

## 12.2 何を作っているの？

ウェブサイトの中で何度も使う部品（入力欄・空っぽ表示・ページ送り・アイコン・パンくず）の **テスト** を強化します。テストとは「コードを書いた後、コンピュータが自動で『ちゃんと動いてる？』を確認してくれる仕組み」のことです。

## 12.3 何が問題だったの？

これまでのテストは「`aria-label` という英語のラベルが付いてるか？」だけを見ていました。これは、車の点検でいうと「ハンドルが付いてるか？」しか見てないようなものです。実際に運転できるかどうかは別の話です。

## 12.4 どう直すの？

`axe-core`（あくす・こあ）という、世界中で使われている「ウェブの使いやすさ自動チェック機」を、テストの中で使えるようにします。これを入れると、「ハンドルが付いてるか」だけじゃなく「ハンドルを回したらタイヤが動くか」まで自動でチェックしてくれます。

## 12.5 用語ミニ辞書

| 用語 | やさしい説明 |
| --- | --- |
| a11y | 「accessibility（アクセシビリティ）」の略。a と y の間に文字が 11 個あるから「a11y」。目が見えない人やキーボードしか使えない人にも使えるか、という話 |
| axe-core | 自動 a11y チェッカー。ページの HTML を見て「ここがダメ」を教えてくれる |
| jest-axe | axe-core を JavaScript の test の中で簡単に呼べるようにしたラッパー |
| jsdom | ブラウザ無しで「擬似的に HTML を動かす」仕組み。test を高速に動かすために使う |
| primitive | 何度も使う UI の最小部品。入力欄やボタンなど |
| rule baseline | axe のチェック項目のうち「これは見てね」「これは無視してね」を決めた一覧 |
| proxy assertion | 「本当に確かめたいことの代わりに、表面だけ確認する」テスト。今回はこれを実検証に置き換える |

## 12.6 なぜ rule の一部を無視するの？

`color-contrast`（色のコントラスト）は、本物のブラウザでないと正しく測れません。テストは速度のために「ブラウザのフリ」（jsdom）で動くので、ここでは無視します。本物の色チェックは Lighthouse や Playwright という別の道具で別途やります。

## 12.7 作業手順（ざっくり）

1. 「みんなで使う axe の設定ファイル」を作る → `apps/web/src/test/axe.ts`
2. 5 つの部品のテストファイルに「axe で検査」を追加する
3. もう要らない「ラベル付いてる？」だけのテストを消す
4. テストを動かして、緑（OK）になることを確認する
5. PR を作って dev ブランチに入れる

## 12.8 影響範囲

- ユーザーに見える画面の変化: **なし**
- ビルドが遅くなる？: ほぼ無し（テスト全体で 2 秒以下の増加）
- 既存機能が壊れる？: テストでしか使わないコードなので、本番には影響しない
