# Phase 2: 最終的一貫性制約と設計指針

## 最終的一貫性の特性

| 項目 | 値 |
| --- | --- |
| 書き込み伝搬時間 | 最大 60 秒（公称値） |
| read-after-write の保証 | なし（同一エッジでも保証されない） |
| グローバル一貫性 | 最終的一貫性（Eventually Consistent） |
| 削除の伝搬 | put と同じく最大 60 秒 |

## 設計指針（AC-7 の根拠）

### 1. put 直後の read を前提にしない

- `SESSION_KV.put(k, v)` 直後に同一キーを `SESSION_KV.get(k)` しても、別エッジ経由なら旧値（または null）が返る可能性がある
- ユーザー操作直後に同じ値を即時 read する設計は禁止

### 2. 即時反映が必要な操作で KV を使わない

| 即時反映が必要な操作 | 採用ストア |
| --- | --- |
| ログアウト即時無効化 | D1（セッションテーブル）または Durable Objects |
| 権限変更（ロール変更等） | D1（ユーザーロールテーブル） |
| 課金状態変更 | D1（請求テーブル） |
| カウンタの正確な集計 | Durable Objects（強整合カウンタ） |

KV はあくまで「最大 60 秒遅れて反映される読み取り中心キャッシュ」と位置づける。

### 3. ブラックリストヒット = 拒否のみ実装

- セッションブラックリストの設計では「ブラックリストヒット = 認証拒否」のみ実装
- 「ブラックリスト未ヒット = 認証許可」では即時失効を保証しない（最大 60 秒間 read-after-write 不整合）
- ログアウト即時失効が必要な場合は、ブラックリスト更新と同時に D1 のセッションフラグを更新し、認証ハンドラは両方を確認する

### 4. TTL 失効後の null チェック

- TTL 失効後の `get` は null を返す（または "Key not found"）
- 全 read 箇所で null チェックを必須化、TTL 失効時のフォールバック（D1 から再構築）を実装ガイドに含める

### 5. 削除（delete）の即時保証なし

- `SESSION_KV.delete(k)` 後も最大 60 秒間は別エッジから旧値が読めるケースを考慮
- 機密データを KV から削除しても、即時に消える保証はない（機密データは KV に置かない設計に変更すべき）

## 違反検出（コードレビュー観点）

レビューで以下のパターンを検出して却下する：

```ts
// 違反: put 直後 get
await env.SESSION_KV.put("k", "v")
const v = await env.SESSION_KV.get("k")  // ← 旧値の可能性

// 違反: 即時失効を期待した delete
await env.SESSION_KV.delete("session:active:" + userId)
// ← 60 秒間は他エッジから読める
```

```ts
// 推奨: ブラックリストは「ヒット = 拒否」のみ
const isBlacklisted = await env.SESSION_KV.get(`session:blacklist:${jti}`)
if (isBlacklisted) {
  return new Response("Forbidden", { status: 403 })
}
// ← 未ヒットでも認証許可は別途 D1 で検証
```
