# Phase 2: Cloudflare KV 無料枠運用方針

## 無料枠制約

| 制約項目 | 上限 | 影響 |
| --- | --- | --- |
| read / day | 100,000 | 設定キャッシュ・セッションブラックリスト読み取りの総量を制限 |
| write / day | 1,000 | セッション・ブラックリスト・レートリミット書き込みの総量を制限 |
| storage | 1 GB | キー総数 × 平均バリューサイズで試算 |
| 1 key の最大サイズ | 25 MB | 通常用途で問題にならない |
| バルク操作 | 10,000 keys / call | 一括削除・移行時に注意 |

## 運用方針

### 書き込み枠（1,000 / day）の確保

- セッション本体を KV に保管しない（JWT のみで完結させる）
- ブラックリスト書き込みのみ許可（ログアウト時のみ）。アクティブログアウトが日次 1,000 件を超える規模では Durable Objects への移行を検討
- 設定キャッシュへの書き込みは設定変更時のみ。日次更新頻度が低い前提
- レートリミット書き込みは TTL 60 秒〜10 分の短期ウィンドウで運用。ウィンドウ更新頻度を運用で監視

### 読み取り枠（100,000 / day）の確保

- 同一キーへの read はワーカー側でメモリキャッシュで吸収する設計を推奨（実装は下流タスク）
- 読み取り集中ポイント（設定キャッシュ）は TTL 1h で頻度を抑制
- セッションブラックリスト読み取りは認証エンドポイント呼び出し回数に比例。100k/day の前提でユーザー規模を見積もる

### ストレージ枠（1 GB）の確保

- セッションブラックリスト: jti（数十バイト）× アクティブセッション数（TTL 24h で上限制御）
- 設定キャッシュ: 設定キー数 × 平均バリューサイズ。MVP 規模では数 MB 以内
- レートリミット: バケット数 × ウィンドウ。短期 TTL で自動的に減容

## 監視・アラート方針

| 項目 | 監視手段 | 閾値（推奨） |
| --- | --- | --- |
| write / day | Cloudflare Analytics | 700/day（70%） で警告、900/day（90%）で対応 |
| read / day | Cloudflare Analytics | 70,000/day（70%）で警告 |
| storage | Cloudflare Dashboard KV メトリクス | 700 MB（70%）で警告 |

監視・アラート実装は本タスクのスコープ外。下流の運用タスクで Cloudflare Analytics への接続を実施する。

## 無料枠枯渇時のフォールバック

| 枯渇種別 | 即時対応 | 中期対応 |
| --- | --- | --- |
| write 枯渇 | 書き込み停止（ブラックリストのみ生かし、レートリミット一時停止） | 用途縮退・Durable Objects への移行 |
| read 枯渇 | 読み取りキャッシュをワーカーメモリに退避 | 読み取り集中ポイントの設計見直し |
| storage 枯渇 | 古い key を bulk delete（TTL 経過済みは自動削除） | ストレージ集中用途の D1 移行検討 |
