# Phase 12: 中学生レベル概念解説（教育用）

[実装区分: 実装仕様書（教育用ドキュメント）]

このフェーズは、task-14 で扱う技術用語・仕組みを **専門知識のない人**（中学生レベル）にも分かるように説明することを目的とする。

## 1. このタスクは何をするの？

UBM 兵庫支部会の会員向けサイトには「**マイページ**」（`/profile`）という画面があります。
ここは「自分のプロフィールが今どう見えているか」「公開範囲を変えたい」「退会したい」を会員自身が確認・申請するための場所です。

今のマイページは「色やボタンの形がバラバラ」「ボタンを連打すると同じ申請が二重に飛んでしまう」など見た目とロジックの問題があり、これを **きれいに作り直す**のが task-14 のゴールです。

---

## 2. 4 つの領域（4 regions）

マイページを縦に 4 つに区切って、それぞれの責任を明確にします。

```
┌────────────────────────────────────┐
│ ① 公開状態バナー                   │ ← 今のあなたは「公開中 / 会員限定公開 / 非公開」のどれ？を一目で
├────────────────────────────────────┤
│ ② 公開範囲サマリ                   │ ← どの項目が公開で、どれが非公開かをリストで
├────────────────────────────────────┤
│ ③ 申請パネル                       │ ← 「公開範囲を変えたい」「退会したい」のボタン
├────────────────────────────────────┤
│ ④ 削除申請ダイアログ               │ ← 退会申請のための確認画面（ポップアップ）
└────────────────────────────────────┘
```

## 3. 用語の説明

| 用語 | 中学生向け説明 |
|------|---------------|
| **公開状態（publishState）** | あなたのプロフィールが「会員一覧に出るかどうか」の状態。3 種類: 全部公開 / 一部だけ公開 / 何も公開しない。 |
| **認証ゲート（authGateState）** | あなたのアカウントが「使える状態」「規約再同意が必要」「削除待ち」のどれかを表す。 |
| **pending（ペンディング）** | 「申請を出したけど、まだ管理者が OK か NG か返事をしていない」状態。 |
| **トークン（OKLch tokens）** | 色の名前帳。「赤系の警告色」「緑系の OK 色」などをあらかじめ決めておき、コードでは色番号ではなく **名前**で呼ぶしくみ。デザインを揃えやすくなる。 |
| **Server Component / Client Component** | 「サーバーで HTML を作って送る部品」と「ブラウザで動く部品」の違い。データ取得はサーバー、ボタン操作はブラウザ、と役割を分ける。 |
| **fetchAuthed** | 「ログイン中のクッキーを自動で持っていって API に問い合わせる」便利な関数。これを通さないと API は応答しない。 |
| **router.refresh()** | ブラウザに「もう一度サーバーから新しいデータを取ってきて」と頼む命令。申請を出した後、最新の pending 状態を画面に出すために使う。 |
| **a11y（アクセシビリティ）** | 目が見えない人のスクリーンリーダーや、マウスが使えない人のキーボード操作でも、画面が正しく使えるようにする仕組み。 |
| **focus trap** | ポップアップ（Dialog）を開いたとき、Tab キーがポップアップの外に逃げないようにする仕掛け。 |
| **IME（Input Method Editor）** | 日本語入力のしくみ。「あ → 亜」のように変換中はまだ確定していない。確定前にボタンが押されないようにする工夫が必要。 |

## 4. なぜ「楽観的 UI」を使わないの？

普通の SNS なら「いいねボタンを押した瞬間にハートを赤く塗る」のが快適です（**楽観的 UI**）。しかしこのサイトの「申請」は **管理者が承認するまで意味を持たない**ため、勝手に「申請済み」と表示すると会員に誤解を与えます。

そこで本タスクでは:
1. ボタンを押す
2. サーバーに申請を送る
3. サーバーの応答を待つ
4. **画面を最新データで描き直す**（router.refresh）

という遠回りに見える作りを **あえて**選びます。

---

## 5. なぜ「本人が直接プロフィールを編集」できないの？

UBM 兵庫支部会では、プロフィールの正本（オリジナルデータ）は **Google Form の回答**です。
本人が画面で勝手に書き換えると、Form のデータと食い違ってしまいます。
そのため MVP（Minimum Viable Product = 最小機能版）では、**プロフィール本文の編集は Google Form を再回答してもらう**設計です。
このマイページではあくまで「**確認**」と「**申請**」だけを行います。

---

## 6. 触らないこと（不変条件）

このタスクで **してはいけないこと**:

1. API（サーバー側のエンドポイント）を新しく追加・変更しない
2. データベースの構造を変えない
3. プロフィール本文を直接編集する UI を作らない
4. 色の番号（HEX、例: `#FF0000`）をコードに直書きしない
5. ブラウザから直接データベースを触らない（必ず API 経由）
6. ボタンを押した瞬間に「申請済み」と表示しない（楽観的 UI 不採用）

---

## 7. このタスクの「完了」とは？

7 つの確認すべてに OK が出る状態です:

1. 4 つの領域がきれいに表示されている
2. 色番号が直書きされていない（tokens を使っている）
3. ペンディング中はボタンがグレーアウトしている
4. 申請ボタンを押すと API に 1 回だけ送信される
5. 削除申請の確認入力が一致するまで送信ボタンが押せない
6. 目の見えない人がスクリーンリーダーで操作できる
7. ログインしていない人は自動でログイン画面に飛ばされる

## 8. 全体の流れ（ユーザー視点）

```
[会員がブラウザで /profile を開く]
   │
   ├─ ログインしていない → ログイン画面へ案内
   │
   └─ ログイン済み
        │
        ├─ ① 公開状態バナーが表示される（緑＝公開 / 青＝一部 / 黄＝非公開 / 赤＝削除待ち）
        ├─ ② どの項目が公開・非公開かリスト表示
        ├─ ③ 申請ボタン（ペンディング中なら押せない）
        │     │
        │     ├─ 「公開範囲変更」を押す → ポップアップ → 選択して送信 → 一覧に「申請中」と表示
        │     └─ 「アカウント削除」を押す → ポップアップ → 確認入力 → 送信 → 一覧に「削除申請中」と表示
        │
        └─ ④ 削除申請ダイアログ（③ から開く）
```
