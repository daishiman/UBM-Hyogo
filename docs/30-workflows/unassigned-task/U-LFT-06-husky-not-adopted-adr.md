## 1. なぜこのタスクが必要か（Why）

### 1.1 背景

Git hooks 管理ツールとして lefthook を採用したが、対抗候補だった husky を採用しなかった理由は lessons L-LH の申し送り B-2 に断片的に残るのみで、ADR としての正本ドキュメントが存在しない。

### 1.2 問題点・課題

- 将来的に「husky でも良いのでは」という再提案が出た際、過去判断の根拠を都度発掘するコストが発生する。
- monorepo / pnpm / 並列 worktree 運用との相性差を文書化しないと再評価が場当たり的になる。

### 1.3 放置した場合の影響

- 同種の意思決定が再発する。
- 新規参加者（将来）が選定意図を読み解けない。

---

## 2. 何を達成するか（What）

### 2.1 目的

husky 不採用 / lefthook 採用の経緯と判断軸を ADR として明文化する。

### 2.2 最終ゴール

- ADR 1 本が作成され、評価軸（並列実行・monorepo 親和性・YAML 設定の正本性 等）と判断結果が記述済み。

### 2.3 スコープ

#### 含むもの

- husky / lefthook の比較ポイント整理
- 採否理由
- 再評価トリガ（「これが起きたら見直す」条件）

#### 含まないもの

- 実装変更（lefthook を覆す作業はしない）

### 2.4 成果物

- ADR ファイル

---

## 3. どのように実行するか（How）

### 3.1 前提条件

- `lefthook.yml` が正本として運用中
- lessons L-LH 申し送り B-2 を参照可能

### 3.2 依存タスク

- なし

### 3.3 推奨アプローチ

1. lessons B-2 の記述を起点に判断軸を抽出。
2. ADR テンプレに沿って状況・選択肢・決定・帰結を記述。
3. PR で merge。
