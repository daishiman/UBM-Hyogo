# Phase 2: 用途別 TTL 方針

## 用途別 TTL 設計

| 用途 | キー命名例 | TTL | 一貫性要件 | KV 採用判断 |
| --- | --- | --- | --- | --- |
| セッションブラックリスト（JWT 失効済み jti） | `session:blacklist:<jti>` | 24h（JWT 有効期限と一致） | 最終的一貫性可（最大 60s 遅延を許容） | 採用 |
| 設定キャッシュ（読み取り中心） | `config:<key>` | 1h（3600 秒） | 最終的一貫性可 | 採用 |
| レートリミットカウンタ（短期） | `rl:<bucket>:<window>` | 60s〜10min | 最終的一貫性可（windowing で吸収） | 採用（書き込み枠監視必須） |
| ログアウト即時反映 | - | - | 強整合性必要 | **不採用**（D1 / Durable Objects で処理） |
| 権限変更即時反映 | - | - | 強整合性必要 | **不採用**（D1 / Durable Objects で処理） |
| セッション本体保管（クッキー代替） | - | - | - | **不採用**（書き込み枠 1k/日 を消費するため。JWT で完結させる） |

## TTL 設計指針

### セッションブラックリスト

- JWT 失効時に `jti` を KV に put、TTL は JWT の `exp` と一致させる
- ブラックリストはログアウト時に書き込まれるため、書き込み頻度はアクティブログアウト数 ≦ 1,000/日 の前提
- ログアウト後最大 60 秒間はブラックリストが伝搬しない可能性があるため、認証ハンドラは「ブラックリストヒット = 拒否」のみ実装し、「ブラックリスト未ヒット = 許可」では即時失効を保証しない

### 設定キャッシュ

- TTL 1h（3600 秒）。設定変更頻度が低く、最大 60 秒の最終的一貫性を許容できる用途のみ
- 書き込みは設定変更時のみ。設定変更が日次 1,000 件を超えることは想定しない
- get 側は null チェック必須、TTL 失効時は D1 から再構築する

### レートリミットカウンタ

- TTL 60 秒〜10 分の短期スライディングウィンドウ
- 書き込み枠を最も消費する可能性が高い用途のため、ウィンドウサイズと書き込み頻度を運用で監視
- write/day の見積もり: アクティブユーザー × リクエスト数 × ウィンドウ更新頻度
- 1,000 write/day を超えるシナリオでは Durable Objects への移行を検討（Phase 6 FC-02 の mitigation）

## 最小 TTL の制約

- KV `cacheTtl` の最小値は 60 秒（Cloudflare 仕様）
- 最終的一貫性の最大伝搬時間（60 秒）以下に TTL を設定しない（伝搬完了前に失効してしまうため）
- 検証用途（短時間 put → get）でも 60 秒以上を推奨
