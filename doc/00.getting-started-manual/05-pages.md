# ページ構成・画面設計

## ルーティング一覧

```text
/                       トップ・ログイン案内
/login                  Googleログイン + メールリンク
/no-access              未登録 / 規約未同意 / 削除済みの案内
/members                メンバー一覧
/members/[id]           メンバー詳細
/profile                自分のプロフィール確認
/profile/edit           自分のプロフィール編集
/admin/members          管理者: 公開状態・削除状態の管理
/admin/schema           管理者: Google Form 変更差分・同期状態
/admin/tags             管理者: タグ辞書・ルール管理
```

---

## 各ページの役割

### `/`

- 未ログインユーザーの入口
- サイトの目的、公開範囲、利用条件を短く表示
- 主導線は「Googleでログイン」
- 補助導線として「メールリンクを受け取る」を表示

### `/login`

- Auth.js v5 のログイン開始ページ
- Googleログインを主導線にする
- メールリンク送信フォームを補助導線にする
- ログイン後はフォーム回答済み判定を行い、未回答は `/no-access` へ送る

### `/no-access`

- フォーム未回答、規約未同意、削除済みメンバー向けの案内
- フォームURL、問い合わせ先、再ログイン導線を表示

### `/members`

- 公開対象かつ非表示化されていないメンバーのみ表示
- 検索対象は名前・職業・ビジネス概要・スキル・自己紹介・タグ
- フィルターは UBM区画 / UBM参加ステータス / 公開状態 / 業種タグ
- 一覧カードは「誰か」がすぐ分かることを優先し、補助情報は2行までに抑える

### `/members/[id]`

- 1人の情報を最短で理解するための詳細ページ
- Hero に名前、肩書き、活動概要、UBM属性、タグを集約する
- 本文は `概要 / ビジネス / つながる / 人となり` の順で表示
- SNS と連絡先はラベル付きチップでまとめる
- 管理者のみ「公開/非公開」「削除/復元」導線を表示する

### `/profile`

- ログイン中ユーザーの自分用ページ
- `members/[id]` と同じ表示基盤を使う
- 非公開項目も本人は閲覧できる
- 編集前提の項目は `profile/edit` に分ける

### `/profile/edit`

- 本人のみアクセス可能
- 編集可能なフィールドだけをフォームに出す
- 保存後は `/profile` に戻す

### `/admin/members`

- 管理者の主画面
- 全メンバーの公開状態と削除状態を管理する
- ここでできる操作は以下のみ
  - 公開/非公開の切り替え
  - 削除/復元
  - 運用メモの確認
- 他人のプロフィール本文編集、タグ辞書管理はデフォルトUXから外す

### `/admin/schema`

- Google Form の構造変化を確認するページ
- 新規質問、削除、質問文変更、選択肢変更、`questionId` 差し替えを一覧で見る
- `extraFields` / `unmappedQuestionIds` / `schemaHash` / `revisionId` を表示する
- 管理者が `stableKey` を採番する導線をここに集約する

### `/admin/tags`

- `tag_definitions` と `tag_rules` を管理するページ
- ルールベース分類の閾値、対象 `stableKey`、優先度を編集する
- ここでも他人の本文は編集しない

## 管理者の差分確認UI

### schema diff

```text
/admin/schema

┌────────────────────────────────────────────────────┐
│ Google Form 変更                              [同期] │
├────────────────────────────────────────────────────┤
│ 追加:  会社紹介文                                  │
│ 変更:  ビジネス概要の文言                         │
│ 削除:  旧質問A                                     │
│ 未解決: q_19f3...                                  │
└────────────────────────────────────────────────────┘
```

### member control

```text
/admin/members

┌────────────────────────────────────────────────────┐
│ 山田 太郎      公開中      [非公開] [削除] [詳細]   │
├────────────────────────────────────────────────────┤
│ 概要 / ビジネス / つながる / 人となり の確認だけ   │
└────────────────────────────────────────────────────┘
```

- 変更は確認ダイアログを通す
- 逆操作があるものは `undo` を出す
- 管理操作は一覧の中に埋め込まず、右ドロワーまたは詳細ペインで完結させる

## 画面遷移図

```text
[未ログイン]
    → /
    → /login
          ↓ Googleログイン成功
    フォーム回答済み？
    ├─ YES → /members
    └─ NO  → /no-access

[ログイン済み・一般メンバー]
    /members → /members/[id]
    /profile → /profile/edit → 保存 → /profile

[ログイン済み・管理者]
    上記すべて
    /admin/members → 公開状態変更 / 削除復元
    /admin/schema → Google Form 差分の確認と再同期
    /admin/tags → タグルール編集
```

## コンポーネント構成

| `SiteHeader` | ナビゲーション・ログアウト・同期状態 |

| `TagRuleEditor` | タグルール編集 |

## 1ユーザーのプロフィール画面で重視すること

1. まず名前と役割が見えること
2. 次に「何をしている人か」が 3 秒で分かること
3. 公開/非公開の判定はページ下部ではなくヘッダー近くで示すこと
4. 連絡先はまとめて触れ、SNS はアイコンだけに依存しないこと
5. 本人用画面と管理者用画面の操作を混ぜないこと
6. 別ドメイン情報は載せないこと
